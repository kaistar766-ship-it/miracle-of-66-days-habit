# Claude Code 단계별 구현 프롬프트: 할 일 관리 앱

> **사용 방법**
> 1. 프로젝트 폴더를 만들고 `todo-app-PRD.md`를 폴더 루트에 넣습니다.
> 2. 해당 폴더에서 Claude Code를 실행합니다.
> 3. 아래 프롬프트를 **1단계부터 순서대로** 하나씩 붙여넣습니다.
> 4. 각 단계의 "완료 확인" 항목을 브라우저에서 직접 검증한 후 다음 단계로 넘어갑니다.
>
> 각 프롬프트는 이전 단계의 결과물 위에 쌓이는 구조입니다. 중간에 문제가 있으면 다음 단계로 넘어가지 말고 같은 단계에서 수정을 요청하세요.

## 1단계: 프로젝트 골격 + 데이터 레이어

```
루트에 있는 todo-app-PRD.md를 읽고 프로젝트 컨텍스트를 파악해 줘.

이번 단계에서는 앱의 골격과 데이터 레이어만 구현한다. UI 디자인은 아직 하지 않는다.

[작업 내용]
1. 파일 구조 생성: index.html, style.css, app.js 3파일 구성.
   빌드 도구, 프레임워크, 외부 라이브러리는 절대 사용하지 말 것. 순수 Vanilla JS만 사용.
   index.html을 더블클릭으로 열었을 때 바로 동작해야 한다 (file:// 프로토콜 호환).

2. app.js에 localStorage 기반 데이터 레이어를 구현:
   - 저장 키: "todoapp.todos" (할 일 배열), "todoapp.challenge" (챌린지 상태)
   - 데이터 스키마는 PRD 6장의 데이터 모델을 그대로 따를 것
   - 함수: loadTodos(), saveTodos(todos), loadChallenge(), saveChallenge(challenge)
   - JSON.parse 실패, localStorage 접근 불가(용량 초과 등) 시 try-catch로 처리하고
     사용자에게 안내 메시지를 표시할 수 있는 훅을 남겨둘 것 (PRD NFR-5)
   - 최초 실행 시 challenge 기본값: targetDays 30, rule "min_count", minCount 1,
     currentStreak 0, bestStreak 0, lastAchievedDate null, history []

3. 할 일 CRUD 순수 함수 구현 (UI와 분리):
   - addTodo(text, category), updateTodo(id, changes), deleteTodo(id), toggleDone(id)
   - id는 Date.now() 기반 문자열, createdAt은 로컬 시간 기준 YYYY-MM-DD
   - 모든 변경 후 자동으로 saveTodos 호출

4. index.html에는 최소한의 뼈대만: 헤더, 진행률 영역, 입력 영역, 필터 탭,
   할 일 목록, 챌린지 영역의 빈 컨테이너를 시맨틱하게 배치 (PRD 7장 순서대로).

[완료 확인]
- 브라우저 콘솔에서 addTodo("테스트", "work") 실행 후 새로고침해도
  localStorage에 데이터가 남아 있어야 한다.
- 콘솔 에러가 없어야 한다.
```

## 2단계: 할 일 CRUD UI + 완료 체크

```
todo-app-PRD.md의 FR-1(CRUD), FR-2(완료 체크)를 구현할 차례다.
1단계에서 만든 데이터 레이어 함수를 그대로 사용하고, 렌더링 로직만 추가해 줘.

[작업 내용]
1. 할 일 추가 (FR-1.1, FR-1.2):
   - 텍스트 입력창 + 카테고리 선택(업무/개인/공부) + 추가 버튼
   - Enter 키로도 추가 가능
   - 빈 문자열/공백만 입력 시 추가되지 않아야 함
   - 카테고리 기본값은 마지막으로 사용한 카테고리 (localStorage에 별도 저장)

2. 목록 렌더링:
   - renderTodos() 단일 함수로 목록 전체를 다시 그리는 방식 (상태 → 화면 단방향)
   - 오늘 날짜(createdAt이 오늘)인 항목만 표시 (FR-1.5)
   - 미완료 항목이 위, 완료 항목이 아래 (FR-2.2)

3. 완료 체크 (FR-2.1, FR-2.3):
   - 체크박스 클릭으로 done 토글
   - 완료 시 completedAt에 ISO timestamp 기록, 해제 시 null
   - 완료 항목은 취소선 + 흐린 색상으로 표시

4. 수정 (FR-1.3):
   - 수정 버튼 클릭 시 해당 항목이 인라인 편집 모드로 전환
     (텍스트 input + 카테고리 select + 저장/취소 버튼)
   - Enter로 저장, Esc로 취소

5. 삭제 (FR-1.4):
   - 삭제 버튼 클릭 시 즉시 삭제하되, 화면 상단에 "삭제됨 [실행 취소]"
     토스트를 5초간 표시. 실행 취소 클릭 시 원래 위치로 복원.

[완료 확인]
- 추가 → 수정 → 완료 체크 → 삭제 → 실행 취소 전체 흐름이 동작해야 한다.
- 각 동작 후 새로고침해도 상태가 유지되어야 한다.
- 이벤트 리스너는 목록 컨테이너에 위임(delegation) 방식으로 붙여서
  렌더링 때마다 리스너가 중복 등록되지 않게 할 것.
```

## 3단계: 카테고리 필터 + 진행률

```
todo-app-PRD.md의 FR-3(카테고리), FR-4(진행률)를 구현할 차례다.

[작업 내용]
1. 카테고리 시각화 (FR-3.2):
   - 각 할 일 항목에 카테고리 색상 태그 표시
   - 색상: 업무(파랑 계열), 개인(초록 계열), 공부(보라 계열) — CSS 변수로 정의

2. 필터 탭 (FR-3.3):
   - 전체 / 업무 / 개인 / 공부 4개 탭
   - 선택된 탭은 시각적으로 구분, 현재 필터 상태는 메모리에만 유지 (저장 불필요)
   - 필터는 목록 표시에만 적용하고, 진행률 계산은 항상 전체 기준

3. 전체 진행률 (FR-4.1, FR-4.3):
   - 헤더 아래에 "완료 N / 전체 M (XX%)" 텍스트 + 가로 진행 바
   - 할 일이 0개일 때는 "오늘의 할 일을 추가해 보세요" 문구 표시 (0% 바 대신)
   - 체크/해제/추가/삭제 시 즉시 갱신

4. 카테고리별 진행률 (FR-4.2):
   - 전체 진행 바 아래에 카테고리 3개의 미니 진행률 표시
     (예: "업무 3/5" + 소형 바), 카테고리 색상과 일치시킬 것
   - 해당 카테고리에 할 일이 없으면 흐리게 표시

[완료 확인]
- 필터를 바꿔도 진행률 수치는 변하지 않아야 한다 (전체 기준 유지).
- 완료 체크 시 전체 진행 바와 해당 카테고리 바가 동시에 갱신되어야 한다.
```

## 4단계: 연속 달성 챌린지 (Streak) 로직

```
todo-app-PRD.md의 FR-5(연속 달성 챌린지)를 구현할 차례다.
이 단계가 앱의 핵심 차별점이므로 날짜 판정 로직을 특히 꼼꼼하게 작성해 줘.

[작업 내용]
1. 날짜 유틸리티:
   - todayStr(): 로컬 시간 기준 YYYY-MM-DD 반환 (UTC가 아닌 로컬 기준임에 주의)
   - dayDiff(dateA, dateB): 두 날짜 문자열 간 일수 차이

2. 달성 판정 (FR-5.2):
   - isTodayAchieved(todos, challenge):
     rule이 "all_done"이면 오늘 할 일이 1개 이상이면서 전부 완료일 때 true,
     rule이 "min_count"이면 오늘 완료 수 >= minCount일 때 true

3. streak 갱신 로직 (FR-5.4, PRD 4장 날짜 판정 규칙):
   - checkStreakOnLoad(): 앱 실행 시 1회 호출.
     lastAchievedDate와 오늘의 차이가 2일 이상이면 currentStreak를 0으로 초기화
     (어제 달성 실패 확정). bestStreak는 절대 건드리지 않는다.
   - markAchievedIfNeeded(): 완료 체크 이벤트마다 호출.
     오늘이 아직 미달성 상태에서 달성 조건을 충족하면:
     currentStreak +1, lastAchievedDate = 오늘, history에 오늘 추가,
     bestStreak = max(bestStreak, currentStreak), 저장.
   - 주의: 체크 해제로 조건이 다시 미달이 되는 경우의 롤백 처리도 구현할 것
     (오늘 이미 달성 처리했는데 해제로 미달이 되면 streak -1, history에서 오늘 제거)
   - history는 최근 60일만 보관, 초과분은 오래된 것부터 제거

4. 챌린지 UI (FR-5.1, FR-5.3):
   - 헤더에 요약 배지: "🔥 12일 연속" 형태
   - 챌린지 영역에 표시: 현재 연속일 / 목표일 진행 바, 최고 기록
   - 설정 버튼 → 목표일 수(자유 입력, 1 이상 정수), 달성 기준(전부 완료 / 최소 N개 완료),
     N값을 변경할 수 있는 설정 폼. 저장 시 즉시 반영.
   - 설정 변경으로 오늘 판정 결과가 달라질 수 있으므로 저장 직후
     markAchievedIfNeeded / 롤백 로직을 재실행할 것

5. 목표 달성 (FR-5.6):
   - currentStreak가 targetDays에 도달하면 축하 메시지 표시 +
     "새 챌린지 설정" 버튼 노출

[완료 확인]
- 콘솔에서 lastAchievedDate를 3일 전으로 조작 후 새로고침 → streak가 0이 되어야 한다.
- 오늘 첫 완료 체크 시 streak +1, 그 체크를 해제하면 다시 -1 되어야 한다.
- bestStreak는 어떤 경우에도 감소하지 않아야 한다.
- 위 시나리오를 검증하는 간단한 테스트 케이스를 주석 또는 별도 함수로 남겨 줘.
```

## 5단계: 히트맵 + 반응형 + 최종 마무리

```
todo-app-PRD.md의 남은 P1 항목과 비기능 요구사항을 마무리할 차례다.

[작업 내용]
1. 달성 히트맵 (FR-5.5):
   - 챌린지 영역에 최근 30일을 격자(7열 x 5행 내외)로 표시
   - challenge.history에 있는 날짜는 진한 색, 없는 날짜는 옅은 회색, 오늘은 테두리 강조
   - 각 칸에 hover/tap 시 날짜 툴팁 (title 속성으로 충분)
   - 외부 라이브러리 없이 순수 CSS grid로 구현

2. 반응형 레이아웃 (NFR-6):
   - 데스크톱: 최대 너비 640px 중앙 정렬
   - 모바일(<480px): 입력 영역 세로 배치, 터치 타깃 최소 44px

3. 에러 안내 (NFR-5):
   - 1단계에서 남겨둔 저장 실패 훅을 실제 토스트 알림으로 연결
   - "저장에 실패했습니다. 브라우저 저장 공간을 확인해 주세요." 형태

4. 최종 점검 및 정리:
   - PRD의 P0/P1 요구사항 전체를 체크리스트로 만들어 코드와 대조하고,
     누락되거나 PRD와 다르게 구현된 부분이 있으면 보고해 줘
   - 콘솔 로그, 사용하지 않는 코드 제거
   - app.js 상단에 파일 구조와 주요 함수 설명 주석 추가
   - README.md 생성: 실행 방법(index.html 더블클릭), 기능 목록,
     데이터 저장 방식과 한계(브라우저/기기 종속) 안내

[완료 확인]
- file:// 로 열어서 전체 기능(추가/수정/삭제/체크/필터/진행률/streak/히트맵)이
  콘솔 에러 없이 동작해야 한다.
- 브라우저 개발자 도구의 모바일 뷰(375px)에서 레이아웃이 깨지지 않아야 한다.
- PRD 대조 체크리스트 결과를 마지막에 출력해 줘.
```

## 부록: 단계별 검증 팁

| 단계 | 사람이 직접 확인할 핵심 1가지 |
|------|------------------------------|
| 1 | 새로고침 후 콘솔에서 `localStorage.getItem("todoapp.todos")` 값 존재 |
| 2 | 삭제 → 실행 취소가 원래 위치로 복원되는지 |
| 3 | 필터 전환 시 진행률 숫자가 변하지 않는지 |
| 4 | 완료 체크 해제 시 streak 롤백이 되는지 |
| 5 | 모바일 뷰(375px)에서 히트맵과 입력 영역 레이아웃 |
